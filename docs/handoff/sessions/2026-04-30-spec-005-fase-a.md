## Decisões tomadas

| Decisão | Onde | Racional |
|---------|------|----------|
| **Plain SQL** ao invés de Alembic para a migration | `api/migrations/003_workflow_canvas_v2.sql` | Repo já usa plain SQL (`001_knowledge_tables.sql`, `002_workflow_tables.sql`); adicionar Alembic agora seria scope creep. |
| **`workflow_edges.source_step_id` como VARCHAR(100)** (não UUID FK) | migration + ORM | Não existe tabela `workflow_steps` neste repo; steps moram em `workflows.definition` JSONB. CASCADE funciona via `workflow_id → workflows(id)`. |
| **`position_x/y` + `merge_policy` em JSONB** (não SQL columns) | migration + schemas | Mantém steps como JSONB (consistência); validação app-side em Phase B `validator.py`. |
| **`updated_by` como VARCHAR(255)** | migration + ORM | Bate com convention `created_by VARCHAR(255)`. Migra para UUID FK quando `users` table existir (TODO genérico do repo). |
| **ESLint rule entregue em 2 formatos** | eslint-rules/ + scripts/check-canvas-imports.sh | ESLint 8 não suporta `rulePaths` em config-file e `next lint` não expõe `--rulesdir`. Plugin standalone fica como instalable; bash script como enforcement de hoje. |
| **Bundle baseline vazio na Fase A** | `bundle-baseline.json` | Baseline real exige Fase E TASK-E05 após canvas existir. Script tolera baseline vazio até lá (mensagem WARN, não FAIL). |
| **Endpoints stubs validam `_require_workflow` ANTES de 501** | `api/workflows/router.py` | Mantém caixa-preta consistente: usuário sem workflow vê 404, com workflow vê 501. Não vaza existência. |

## Aprendizados / pegadinhas

- **Spec assumiu Alembic; repo usa plain SQL.** Sempre validar pressupostos do design.md vs. realidade do repo antes de iniciar Phase A. Faria diferente: TASK-A02 do design (criar SQLAlchemy ORM `WorkflowStep`) virou no-op porque steps são JSONB. Vale anotar no design.md como deviation.
- **Não havia tabela `workflow_steps` no repo.** Decisão de manter steps em JSONB foi do SPEC-003 e mantida; vale documentar no constitution.md futura como invariante explícito ("steps são JSONB; edges são tabela; coordenadas são JSONB"), porque isso vira armadilha em Phase B (validator precisa parsear JSONB para validar `merge_policy`).
- **ESLint custom rule + Next 14 + ESLint 8 = atrito.** `rulePaths` não existe em config-file; `next lint` não tem `--rulesdir`. Solução pragmática: plugin standalone com self-test (`eslint-rules/`) **mais** bash script (`scripts/check-canvas-imports.sh`) que sempre roda. Plugin documenta a regra; script enforce. Vale promover esse pattern para futuras regras estruturais do repo.
- **Caixa-preta nos endpoints stub.** Stub que retorna 501 sem validar workflow existência vaza informação ("workflow id sem RBAC vê 501 vs. 404 — sabe que existe"). Helper `_require_workflow` resolve. Pattern reusável: **stubs sempre validam acesso primeiro, comportamento depois**.
- **Negative test do enforcement script é load-bearing.** Plantar import forbidden + rodar script + verificar exit 1 deu confiança real de que o script funciona quando precisar. Regression test obvio em retrospecto, vale incluir em CI.
- **Test fixtures como contract.** Os 3 seed fixtures (`v1_legacy`, `v2_linear`, `v2_fanout_merge`) ficam como contract da forma esperada do JSONB para Fase B. Cualquier mudança no shape do step JSON força mudança nas fixtures — tag de canary natural.
- **Bundle baseline vazio na Fase A é OK.** Baseline real exige Fase E (canvas real existe). Script tolera baseline vazio com WARN — não fail-closed. Decisão consciente para evitar gating Fase A por uma medida que ainda não tem sentido.
