## Decisões tomadas

| Decisão | Onde | Racional |
|---------|------|----------|
| **`@xyflow/react@^12`** ao invés de `reactflow` literal | package.json | reactflow v12 foi renomeado para @xyflow/react. ESLint rule + canvas-imports check cobrem ambos os nomes. |
| **`NodeShell` helper compartilhado** | nodes/NodeShell.tsx | Cortou ~70% de duplicação visual entre os 7 nodes. Cada node fica responsável só de cor + handles + ícone. |
| **Mock-mode degradation explícita** | WorkflowCanvas | Canvas precisa funcionar sem `NEXT_PUBLIC_API_URL` (estado atual da maioria das telas). Edges ficam apenas em UI; persistência local via context. |
| **2 canais de auto-save (steps + edges)** | WorkflowCanvas | Mexer só em positions não dispara PUT de steps; mexer só em edges não dispara PUT. Banner mostra o pior dos dois. |
| **Migration JIT em duas frentes (server + frontend mock)** | page.tsx + buildEdgesFromV1 helper | Real-mode chama `/migrate-v2`; mock-mode replica a lógica do server (mesma decisão de handle: `next_step→out`, condition→`then`/`else`, hitl→`approved`). |
| **Mobile read-only inline (em vez de componente separado)** | WorkflowCanvas | Detector + banner são pequenos; separar geraria duplicação de prop drilling. Documentado em comentário top-of-file. |
| **`useWorkflowAutoSave` race-safe via useRef + inFlight** | hooks/useWorkflowAutoSave.ts | Resolve I6 da revisão crítica da SPEC: re-fire automático quando edits mid-save mudam o snapshot. |
| **`isValidConnection` server-parity check** | useGraphValidation | Mesma matriz `ALLOWED_SOURCE_HANDLES_BY_TYPE` que `api/workflows/validator.py` — duplicação intencional, paridade simbólica. |
| **`useAutoLayout` lazy import dagre** | hooks/useAutoLayout.ts | `dagre` importado estático aqui (dentro do canvas/) é fine; lazy load real é via `next/dynamic` na page. |

## Aprendizados / pegadinhas

- **`reactflow@^12` → `@xyflow/react`.** Spec de constituição precisa ser atualizada num próximo refresh (não vital — o nome técnico mudou mas a lib é a mesma).
- **NodeShell padrão é high-leverage.** Cortou ~70% da duplicação dos 7 nodes. Próxima vez que um node-type novo aparecer (V2: Loop?), só precisa cor + handles + ícone. Pattern reusable.
- **Mock-mode degradation foi crítica.** Sem ela, o canvas precisaria do backend rodando para nem renderizar. Com ela, demo + dev experience funciona sem fricção. Vale promover esse pattern (`apiAvailable()` guard inline) como padrão para qualquer feature nova até DB persistence chegar.
- **`isValidConnection` server-parity check.** Mantive matriz `ALLOWED_SOURCE_HANDLES_BY_TYPE` no frontend espelhando a do backend (`api/workflows/validator.py`). Pequeno drift de paridade vai morder cedo: vale criar `lib/canvas-handle-rules.ts` shared-package quando dev e tests crescerem.
- **`useWorkflowAutoSave` race-safe via useRef foi load-bearing.** Implementação ingênua (effect-based) perderia edits durante save — exatamente o bug do I6 da revisão crítica da SPEC-005. Confirmou na prática que o pattern documentado funciona.
- **`@xyflow/react` v12 importa `'@xyflow/react/dist/style.css'`** que é processado pelo Next.js automaticamente. Nada de PostCSS extra. Bem.
- **CSS variables sunOS funcionam dentro do React Flow.** Estilizei tudo via inline styles + `var(--deep)`/`var(--sun)`/etc.; React Flow não impõe Tailwind nem styled-components. Aderência ao design system manteve consistência sem fricção.
- **Mobile read-only inline (em vez de componente separado) foi melhor que esperado.** O detector + banner são ~15 linhas; separar adicionaria prop drilling sem ganho. Documentei no top do `WorkflowCanvas.tsx`. Padrão reusable para detalhes responsivos pequenos.
- **Persistência em 2 canais (steps + edges)** evita PUT desnecessário, mas o banner precisou somar status manualmente — ad-hoc reduce. Se uma 3ª resource entrar (ex: workflow metadata), refatorar pra um `useMultiAutoSave` que aceita N saveFns.
