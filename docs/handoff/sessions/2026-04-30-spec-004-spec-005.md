### Revisão crítica pós-escrita (SPEC-005)

A pedido do Heitor, fiz revisão crítica dos 5 artefatos. Encontrei **17 issues** organizados por severidade:

- **3 bloqueadores:**
  - B1: inconsistência crítica `out` vs `success` (constitution dizia `out` universal mas tabela §4.1 dizia tools/llm têm `success`/`error` — migration usaria handle inválido).
  - B2: 3 bugs reais no pseudocódigo do compiler v2 (entry único hardcoded, error-routing não funciona como descrito, MergeNode `any` sem caminho diferenciado).
  - B3: erro de cópia na DoD da constitution (texto autorreferencial v1 == v1).
- **6 importantes** (NFR perf vs limite 20 nodes, fluxos de erro incompletos, `updated_by` faltando no schema, validator sem `no_entry_node`, PUT sem chamar validator, race condition no `useWorkflowAutoSave`).
- **8 nice-to-fix** (Zustand explanation impreciso, "Ctrl+Z nativo do React Flow" inexistente, tamanhos otimistas, CAs sem implementação clara, MergeNode timeout fantasma, mobile sem task, FR-WBC-15 vago com `output_value` inexistente).

Heitor aprovou opção (a) para B1 (handle `out` universal) e "seguir recomendação" para B2/B3 e implícito para todos. Apliquei as 17 correções:
- B1: refatorou todo o sistema de handles. `success` foi removido — `out` é o único handle de saída no caminho feliz; `error` é opcional; `then`/`else` em condition; `approved`/`rejected`/`modified` em hitl. Migration converte `next_step` → `out` para todos os tipos sem special-case.
- B2: pseudocódigo do compiler reescrito com loop sobre `entries` + sorted, error-routing via `add_conditional_edges` + `_route_after_step` lendo marker do state, MergeNode `any` via `_make_merge_any_node` documentado em §6.2 com helpers de evento + cancelamento.
- B3: DoD reescrita: "compiler v2 (lendo edges) produz `StateGraph` estruturalmente equivalente ao compiler v1 (lendo `next_step`/`condition.then|else`)" + definição operacional ("mesma lista de tuplas").
- I3: `ALTER TABLE workflows ADD COLUMN updated_by UUID` adicionado.
- I4: kind `no_entry_node` adicionado em validator + Literal + CAs + §4.3.
- I5: TASK-B01b nova (PUT validation enforcement com subset crítico). 4 CAs novos (CA-43..46).
- I6: `useWorkflowAutoSave` reescrito com `useRef` + `inFlightRef` + comparação pré/pós-save.

## Aprendizados / pegadinhas

- **Brainstorming + sdd-koro skills compõem bem.** Brainstorming drives design decisions via 3 perguntas-chave; sdd-koro drives spec artifact generation. Transição é natural quando decisões estão travadas.
- **Auto mode encurta brainstorming.** Sinaliza "para de perguntar, decide e apresenta". Bundlei as 2 últimas perguntas em defaults razoáveis e perguntei só 3 confirm questions no fim. Heitor respondeu com 1-line corrections — muito mais rápido que 5 sequenciais.
- **Padrão "predecessor SPEC handling".** Quando reverte uma decisão, marcar a SPEC antiga com `status: substituido` + `substituido_por:` + banner em CADA artefato, E na nova SPEC adicionar `predecessor_decision_reversed` no frontmatter apontando linha exata. Permite rastrear evolução sem git archaeology.
- **Self-review pós-escrita pega bugs reais.** SPEC-005 inicialmente passou no placeholder scan (TODOs/XXXs eram todos legítimos), mas a revisão crítica posterior achou 3 bloqueadores não-óbvios: (B1) inconsistência de handles entre constitution/spec/migration, (B2) 3 bugs no pseudocódigo do compiler que não funcionariam como descrito, (B3) erro de cópia na DoD. Lição: placeholder scan não substitui leitura crítica de artefatos cruzados. **Promover "revisão crítica de cross-references" como passo obrigatório do workflow large.**
- **Subagent Explore para extrair material upstream literal.** Para SPEC-004 (FA-13) com 13 docs upstream maduros, despachei Explore com prompt rigoroso pedindo texto literal (sem paráfrase) de cada item específico. Output 84KB salvou ~50K tokens de leitura no main context. Trick: subagent results muito grandes vão para arquivo JSON; use `jq -r '.[0].text' file.json > /tmp/extract.md` + Read em chunks.
- **Auto-save com race condition é fácil de errar.** O design inicial tinha um `useEffect` simples que perderia edits durante save em andamento (status volta a 'dirty' MAS o `useEffect` não re-dispara até próximo `debouncedDefinition` tick). Solução: `useRef` para latest definition + `inFlightRef` + comparação pré/pós-save. Vale lembrar para próximas SPECs com auto-save.
- **Handle universal vs handles tipados é decisão sutil que afeta migration.** Inicialmente eu tinha `success`/`error` como handles separados em tools/llm. A migration v1→v2 ia convertendo `next_step` em handle `success` (porque era o "default verde"). Mas isso forçava a migration a saber o tipo do source para decidir o handle, e o validator de handle precisaria checar tipo. **Decisão B1 (handle `out` universal) eliminou TODOS esses special cases** — `out` funciona em qualquer tipo, migration é trivial, validator simples. Lesson: prefira universal sobre tipado, a menos que tipado dê valor real (não dá quando o "default" é semanticamente o mesmo).
- **`<system-reminder>` sobre TaskList.** A skill envia reminders periódicos sobre TaskCreate/Update se eu não estou usando. Levei a sério durante a sessão (criei tasks #1-17 ao longo do dia), mas em alguns momentos os reminders não eram aplicáveis (durante edits sequenciais de um mesmo artefato). Adverbio: confiar no julgamento sobre quando o tracking adiciona valor.
- **Pivot mid-session sem fechar escopo anterior.** O prompt da sessão era SPEC-004 → revisão → SPEC-005 (drive). Heitor pivotou para canvas no meio. Aceitei e escrevi canvas, mas drive ficou pendente. Lição: ao pivotar, registrar explicitamente "X fica para próxima sessão" no momento do pivot — eu fiz isso ao final, mas seria mais limpo no momento.
- **Reversão de decisão histórica documentada estruturalmente.** SPEC-003 dizia explicitamente "Não é drag-and-drop visual de nodes". SPEC-005 reverte. O frontmatter `predecessor_decision_reversed` apontando linha 81 da constitution antiga + banner em todos os artefatos da SPEC-003 deixa o caminho de evolução visível. **Promover esse pattern para template-large.md.**
